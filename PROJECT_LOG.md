# 项目日志：三线学习训练器

日期：2026-06-29 ~ 06-30

## 项目目标

为日语、英语和《逻辑哲学论》做一个适合个人长期使用的学习程序。核心学习偏好是：

- 少做孤立背诵，多用课文、长句和上下文记忆。
- 练习要有互动感、即时反馈和可见进度。
- 题库和推荐形式要根据每日学习内容、掌握情况和偏好逐步调整。
- 主要使用场景是在 iPhone 上通勤学习，路线为南与野 → 赤羽 → 王子 → 东大前。

## 当前实现

项目是一个静态 Web App，主要文件：

- `index.html`：入口页面和 PWA 元信息。
- `app.js`：题库、状态管理、推荐算法、通勤模式、同步逻辑。
- `styles.css`：桌面和 iPhone 移动端样式。
- `manifest.webmanifest`：PWA 配置。
- `service-worker.js`：HTTPS 环境下缓存核心文件。
- `icon.svg`：主屏幕图标。
- `README.md`：安装、使用、部署和同步说明。

根据实际路线做分段推荐：

通勤队列会跨科目选题，同时考虑今日记录、到期卡、弱项和当前路段适合的题型。

### 6. 手机和电脑数据交互

第一步实现了手动同步：

- 生成备份 JSON。
- 复制或下载备份。
- 在另一台设备导入并合并。

合并策略：

- 自定义题和每日记录按 ID 合并。
- 学习进度保留练习次数更多或更新时间更晚的版本。
- 历史记录按时间合并。

### 7. GitHub 云同步

为了满足电车上脱离局域网使用，加入 GitHub 方案：

- GitHub Pages 用于发布网页。
- 私有 GitHub Gist 用于保存学习进度 JSON。
- 手机和电脑打开同一个页面后，可以自动“拉取合并 → 上传”。

安全处理：

- GitHub token 只保存在当前浏览器的 `localStorage`。
- token 不会写入“生成备份”的 JSON。
- 建议 token 只授予 `gist` 权限。

### 8. 工程优化与可发布性

在功能基本稳定后，做了一轮以"能可靠发布到 iPhone"为目标的优化：

- Service Worker 由 cache-first 改为 network-first（应用外壳）+ cache-first（图标等静态资源）。这样发布新版本到 GitHub Pages 后，手机联网时会自动拿到最新代码，离线时仍回退缓存。
- Service Worker 不再拦截跨域 GET，避免把 GitHub Gist 的同步数据缓存成旧版本。
- 新增 `apple-touch-icon.png`（180×180）并在 `index.html` 中声明，修复 iOS"添加到主屏幕"图标不显示的问题。
- `shuffle` 由 `sort(random)` 改为 Fisher-Yates，消除出题顺序的系统性偏置。
- 增强健壮性：`loadState` 显式处理空/非法存储；今日目标为 0 时不再除零。
- `app.js` 增加环境判断，可在 Node 中被 `require`，并导出纯逻辑函数；新增 `test.js` 单元测试。

### 10. 修复"重新打开后做过的题重复出现"

- 根因：`due` 默认模式与 `buildSmartQueue` / `buildCommuteSmartQueue` 的兜底桶用 `shuffle(pool)`
  把**整副牌**（含已掌握、还没到复习时间的题）补进队列，导致重开后旧题反复出现；
  且"弱项桶"对答错过的卡不看到期时间也会重新拉出来。
- 修复：新增 `isAvailable(card)`（到期或从没见过），让自动模式（今日到期 / 智能推荐 / 通勤）
  的所有桶都只从"可用卡"里取。做完整副牌后重开会进入"已清空"空状态，而不是重复旧题；
  有新题时仍能填满整轮。`混合挑战` / `弱项加练` 作为用户显式选择的模式保持原样。
- 验证：浏览器自动答完日语全部 44 张后重开，智能推荐返回空（无重复）；答题总数从有 bug 时的 192 收敛到 44。

## 关键设计决策

### 为什么不是直接用 Git 命令同步？

iPhone Safari 不能直接运行 `git pull` / `git push`，所以纯 Git 命令不适合移动端网页。更合适的方案是：

- GitHub Pages 提供静态网页访问。
- GitHub Gist API 提供数据同步。

这相当于“用 GitHub 生态解决同步”，但不是让手机浏览器执行 Git 命令。

### 为什么不用局域网作为长期方案？

局域网方案只适合调试。通勤时手机无法访问家里或电脑局域网，因此长期方案必须是公网 HTTPS。GitHub Pages 满足这个条件。

### 为什么 DeepSeek 走本地代理而不是前端直连？

API key 绝不能放在静态前端（GitHub Pages 是公开的，key 会泄露）。所以做成本地代理 `proxy/deepseek-proxy.mjs`：
key 只从环境变量读，只监听 127.0.0.1，前端通过它间接调用。配合"电脑生成、Gist 同步到手机"的流程，
通勤时手机也不需要接触 key。若以后要在电车上即时出题，再把这个代理升级成云端 serverless（key 放云端 secret）。

## iPhone 使用方案

长期推荐：

1. 将项目推送到 GitHub 仓库。
2. 开启 GitHub Pages。
3. 用 iPhone Safari 打开 Pages 地址。
4. 添加到主屏幕。
5. 在“数据同步 → GitHub 云同步”中配置 token。
6. 第一次点击“创建云端”。
7. 保持“联网时自动同步”开启。

通勤使用建议：

1. 上车前或站台先填“今日记录”。
2. 等车/换乘时选“站台/换乘”。
3. 南与野到赤羽做课文长句。
4. 赤羽到王子做短题。
5. 王子到东大前做默写、复述或哲学段落。

6. **题库批量导入**（中等）：支持 CSV / JSON 一次性导入自定义题。
7. **统一空状态 / 加载态轻提示**（简单）：把各处加载/成功/失败统一成 toast 风格（动效与按钮触觉反馈已做）。

**低优先 / 基础设施**

8. **补纯逻辑单元测试**（中等）：把 `buildSmartQueue` / `buildCommuteSmartQueue` / `moduleSpeedToday` 等拆成可测函数（已先补 `shuffledOrder`）。

**已知限制（非 bug）**

- 日语「解析」默认走本地模型，小模型（gemma3:4b）质量差、会跑偏；建议用 `deepseek-r1:8b` 或在面板切 DeepSeek。
- AI 出题 / 解析需要电脑上跑代理，手机端（通勤时）不能现场调用，只能用电脑生成好的结果。

### 17. DeepSeek API 直连 + ChatGPT 剪贴板解析（2026-07-01）

给手机端加了两条不依赖电脑代理的解析/出题/诊断通道：

**DeepSeek API 直连（B 方案）：**
- 前端直接 `fetch("https://api.deepseek.com/chat/completions")`，不经过代理。
- 新增 `deepseekKeyAvailable()` / `callDeepSeekDirect()` / `callDeepSeekAnalyze()` / `callDeepSeekGenerate()` / `callDeepSeekDiagnose()` 等工具函数。
- 解析/出题/诊断都优先走 DeepSeek 直连（如果 key 已设置），回退到本地代理。
- `renderAnalysisBlock` 标注 provider（deepseek-direct / chatgpt / deepseek / 本地模型）。
- 设置面板新增 DeepSeek API key 输入框（密码类型），key 只存 `localStorage`。
- `persistedState()` 主动剥离 `deepseekKey`（和 `gitSync.token` 一样），不进备份/Gist。

**ChatGPT 剪贴板解析（A 方案）：**
- 日语卡片新增「C解析」按钮 → `handleChatGptAnalyze()`：自动构建解析提示词 + 复制到剪贴板 + 跳转 `https://chatgpt.com/`。
- 弹出一个粘贴区（`renderChatGptPasteBlock`）→ 用户把 ChatGPT 返回的结果贴进去 → `handleChatGptPaste()` 保存到 `state.analyses`。
- provider 标注为 `"chatgpt"`。
- 零 API key 依赖，手机/电脑都能用。

**改动的文件：** `app.js`（+~200 行）、`styles.css`（+16 行）。
